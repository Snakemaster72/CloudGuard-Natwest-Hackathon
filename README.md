# CloudGuard: Cloud Spend Intelligence for AWS & GCP

CloudGuard is a cloud spend intelligence MVP for **FinOps-style** visibility and planning. It uses a **reproducible synthetic dataset** (AWS/GCP daily spend by provider/service/region/category) to power **forecasting**, **anomaly detection**, **scenario simulation**, and optional **LLM-grounded explanations**.

Intended users: startup founders, engineering leads, and FinOps analysts who want early warning signals and “what-if” planning without waiting for month-end billing.

## Features (implemented)

- **Filter-safe slicing** by Provider → Service → Region → Category (UI prevents invalid combinations).
- **Forecasting**: daily spend forecast for a selected slice with uncertainty band.
- **Baseline run-rate**: moving-average baseline for quick comparison.
- **Anomaly detection**: flags historical days outside expected range with a short explanation.
- **Scenario simulation**: apply workload multiplier and unit-cost delta from a start date; see overlay + deltas.
- **AI Copilot** (optional): Gemini-powered explanations/chat grounded only in the selected slice + current outputs.
  - If `GEMINI_API_KEY` is missing, CloudGuard returns deterministic fallback explanations.

## Quick start (local, no Docker)

1) Create env file:
- Copy `.env.example` → `.env`
- (Optional) set `GEMINI_API_KEY=...` to enable Copilot explanations

2) Run the stack locally:
- Follow `DEV_LOCAL.md` to start:
  - ML service (FastAPI) on `:8001`
  - Backend (Express) on `:8000`
  - Frontend (Vite) on `:5173`

3) Open:
- Frontend: http://localhost:5173
- Backend health: http://localhost:8000/health
- ML health: http://localhost:8001/health

## Usage (once running)

1) Pick a **Provider** and then a **Service** (service options are provider-valid).
2) Click **Run forecast** to populate:
   - Forecast chart (with uncertainty band)
   - Baseline
   - Anomalies list
3) Use **Scenario simulation**:
   - Set start date (defaults to tomorrow)
   - Set workload multiplier (e.g. 1.2)
   - Set unit-rate delta (e.g. -0.08)
   - Click **Run scenario**
4) Use **Cloud AI copilot**:
   - Click “Explain forecast / anomaly / scenario impact”
   - Or ask a free-form question

### API examples

Backend proxies the ML service.

- Dataset metadata:
  - `GET /api/dataset`
- Forecast for selected dataset slice:
  - `POST /api/forecast_from_dataset` with body `{ "filters": {"provider":"aws","service":"ec2","region":"us-east-1","category":"compute"}, "horizonDays": 30, "intervalWidth": 0.9 }`
- Anomalies for selected dataset slice:
  - `POST /api/anomalies_from_dataset` with body `{ "filters": { ... } }`

## Dataset (reproducible)

The canonical dataset lives at `data/synthetic_cloud_costs.csv` and is generated by `src/data_generator.py`.
It only contains **valid** provider/service/category/region combinations (so slices are realistic and reproducible).

## Tech stack

- Frontend: React + TypeScript + Vite + Tailwind + Recharts
- Backend: Node.js (Express) API gateway + (optional) Google Gemini
- ML service: Python FastAPI + Prophet
- Data: generated CSV dataset in `data/synthetic_cloud_costs.csv`

## Tests

- Python tests live in `tests/` (e.g. forecasting sanity checks).

## Limitations (honest)

- Dataset is synthetic (for reproducibility); it is not connected to live AWS/GCP billing exports.
- Forecasting is univariate per slice; cross-service attribution is not implemented in this MVP.

## Prerequisites

- Node.js **20+** (frontend + backend)
- Python **3.11** recommended (best Prophet wheel support)
- Poetry (for `ml-service/` environment management)

If you prefer not to use Poetry, you can still run the repo-level scripts with `pip` using `requirements.txt`, but the ML service is designed to run via Poetry.
